    2. Create `packages/web/lib/api-keys/generator.ts`:
       - Import randomBytes, createHash from crypto
       - Export generateApiKey(): { raw: string; hash: string; prefix: string }
       - raw = 'bsk_' + randomBytes(32).toString('base64url')
       - prefix = raw.slice(0, 12) (bsk_ + 8 chars for display/lookup index)
       - hash = createHash('sha256').update(raw).digest('hex')

    3. Create `packages/web/lib/api-keys/authenticator.ts`:
       - Import createHash from crypto
       - Export type ApiKeyAuthResult = { valid: true; tenantId: string; permissions: string[] } | { valid: false }
       - Export async authenticateApiKey(supabase, rawKey): Promise<ApiKeyAuthResult>
       - Return { valid: false } if !rawKey.startsWith('bsk_')
       - Compute prefix = rawKey.slice(0, 12), hash = sha256(rawKey)
       - Query api_keys table: .eq('key_prefix', prefix).eq('key_hash', hash).is('revoked_at', null).single()
       - Check expires_at < now() -> { valid: false }
       - Fire-and-forget update last_used_at = new Date().toISOString()
       - Return { valid: true, tenantId: data.tenant_id, permissions: data.permissions }

    4. Create `packages/web/lib/api-keys/rate-limiter.ts`:
       - Export getRateLimiter(): Ratelimit | null
       - Return null if UPSTASH_REDIS_REST_URL env var is absent (no-op for self-hosters)
       - Singleton pattern with module-level variable
       - Sliding window: 100 requests per 1 minute (default; can be overridden per plan)
       - Export checkRateLimit(identifier: string): Promise<{ success: boolean; reset?: number }>
       - If no limiter, return { success: true }
       - Otherwise call limiter.limit(identifier) and return result

    5. Create `packages/web/lib/api-keys/middleware.ts`:
       - Export withApiKeyAuth(handler): wraps a route handler
       - Extract Authorization header, expect "Bearer bsk_..."
       - Call authenticateApiKey
       - If invalid: return 401
       - Call checkRateLimit with key prefix as identifier
       - If rate limited: return 429 with Retry-After header
       - Pass tenantId and permissions to handler via request context

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client -> /api/keys | Authenticated user creating/listing/revoking API keys |
| External app -> withApiKeyAuth | External application authenticating via API key |
| Rate limiter -> Upstash Redis | Network call to external Redis service |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-05 | Spoofing | API key auth | mitigate | SHA-256 hash comparison; prefix-based lookup limits search space; keys are 32 bytes random (2^256 entropy) |
| T-06-06 | Information Disclosure | /api/keys GET | mitigate | Never return key_hash in API responses; raw key shown once at creation only |
| T-06-07 | Denial of Service | API key endpoints | mitigate | Rate limiting via @upstash/ratelimit sliding window; 429 response with Retry-After header |
| T-06-08 | Elevation of Privilege | API key permissions | mitigate | Permissions stored per-key; middleware checks permissions against requested operation |
| T-06-09 | Tampering | API key revocation | mitigate | Keys scoped to tenant; only tenant members can revoke; revocation is permanent (revoked_at is never cleared) |
</threat_model>
